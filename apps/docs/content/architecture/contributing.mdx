## The Prime Directive

**Zero improvisation.** Design decisions are locked in `04-enterstellar-design-choices.md`. Do not change them mid-implementation. If you hit a genuine blocker or a decision that wasn't anticipated, raise it before writing code ŌĆö not after. Ad-hoc changes break strategic moats.

---

## The 3-Step Workflow

Every unit of work follows this order ŌĆö no exceptions:

**1. Plan**

Before writing implementation code for any package, create its `README.md` first (Module README Standard below). Define the API surface area before filling it in. This prevents scope creep and API drift ŌĆö once the README is written, the API is locked for that session.

**2. Execute**

Write strict TypeScript against the Bible's API signatures (`03-enterstellar-implementation-bible.md`). No `any`. No improvisation on types. Reference the Design Choices doc for every non-trivial decision.

**3. Verify**

Run `pnpm turbo run test` and confirm coverage Ōēź90% before marking anything done. For intent pipeline work, run `enterstellar test --intent="your intent"` to prove the pipeline resolves correctly.

## TypeScript Rules

- `strict: true` in every `tsconfig.json`. No overrides.
- No `any` ŌĆö not explicit, not implicit. Use `unknown` with Zod parsing.
- `catch` clauses use `unknown`, narrowed before use.
- All public exports have JSDoc (`/** ... */`).
- All schemas use Zod. No hand-rolled type assertions.

## Error Handling

All Enterstellar packages throw `EnterstellarError extends Error` with three mandatory fields:

```typescript
class EnterstellarError extends Error {
  code: string;       // e.g. 'ENS-2001'
  module: string;     // e.g. '@enterstellar-ai/compiler'
  recoverable: boolean;
}
```

### Error Taxonomy

| Code | Scenario | Recovery |
|:---|:---|:---|
| `ENS-1001` | Duplicate component name in registry | **Fatal** ŌĆö throw immediately at registration time |
| `ENS-2001` | Zod schema parse failure | Self-correct |
| `ENS-2002` | Hallucinated design token | Reject; coerce to nearest valid token (ENS-2007 warning) after retries |
| `ENS-2003` | Missing accessibility attribute | Auto-inject (if `autoAccessibility: true`) |
| `ENS-2004` | Unknown component name from LLM | Self-correct (send error to LLM, retry) |
| `ENS-3001` | Invalid zone config | **Fatal** ŌĆö dev error, throw immediately |
| `ENS-3002` | Agent timeout | Fallback component |
| `ENS-4001` | Forge generation fail | Fallback component |
| `ENS-4007` | State schema major version mismatch | **Fatal** ŌĆö hard reject, no data loss |

### Error Rules

- **LLM errors** ŌåÆ recoverable (self-correct or coerce)
- **Infrastructure errors** ŌåÆ fallback component, log to trace
- **Developer errors** (bad config, invalid zone) ŌåÆ fatal throw
- All error messages must match the exact spec in `03-enterstellar-implementation-bible.md ┬¦4.2`

## Dependency Rules

| Rule | Detail |
|:---|:---|
| **`@enterstellar-ai/types` is always a peer dep** | Every package lists it in `peerDependencies`, not `dependencies` |
| **Engine packages have zero framework deps** | No `react`, `react-dom`, `react-native`, or any framework import in engine packages (L15) |
| **L15 is CI-enforced** | Custom ESLint rule fails CI if any engine package imports a framework dep |
| **Syncpack enforces version consistency** | No two packages may depend on different versions of the same library |

`@enterstellar-ai/react` is the single exception: it lists all engine packages (`compiler`, `registry`, `state`, `telemetry`, `cache`, `lifecycle`, `connection`, `adapters`, `types`) as regular `dependencies` ŌĆö not peer deps. Consumers install `@enterstellar-ai/react` and get the full engine transitively.

## Testing Requirements

Every module ships with tests before it ships features.

| Test Type | Requirement |
|:---|:---|
| **Unit tests** | Ōēź90% line coverage. Every exported function has dedicated tests. |
| **Integration tests** | Each module tested with its direct dependencies (no mocks for internal deps). |
| **Contract tests** | Every `ComponentContract` in `examples/` passes `registry.validate()`. |
| **Compilation tests** | Every `CompilationError` message text matches the exact spec in Bible ┬¦4.2. |
| **Snapshot tests** | DevTools panel renders match approved Storybook snapshots. |

### Running Tests

```bash
# All packages
pnpm turbo run test

# Single package
pnpm --filter @enterstellar-ai/compiler test

# With coverage report
pnpm turbo run test -- --coverage
```

## Versioning

**Package versioning** is independent per package (Babel model), managed with Changesets:

```bash
# Start a changeset for your PR
pnpm changeset

# Releases are handled in CI on merge to main
```

- Never manually bump `package.json` versions ŌĆö always use Changesets.
- `@enterstellar-ai/types` may be `v1.x` while `@enterstellar-ai/forge` is `v0.x`.

**ComponentContract versioning:** Contracts include a `version: string` field (semver). Registry serves latest only ŌĆö no multi-version serving. If Zod parse succeeds ŌåÆ accept. If Zod parse fails (breaking change) ŌåÆ self-correction loop ŌåÆ fallback. Zod is the migration layer.

## Module README Standard

Every package's `README.md` must contain, in order:

1. **Purpose** ŌĆö 1 paragraph. What this package does and why it exists.
2. **Quick Start** ŌĆö minimal working code example.
3. **API Reference** ŌĆö all exported functions and interfaces with parameter descriptions.
4. **Configuration** ŌĆö all config options with defaults and valid values.
5. **Link to Bible** ŌĆö `See [Implementation Bible ┬¦4.x](../../../agent/03-enterstellar-implementation-bible.md)` for canonical spec.

This is the first thing a developer or AI agent reads. If someone needs to open the Bible to understand the README, the README is incomplete.

## Per-PR Checklist

Before opening a PR:

- [ ] `pnpm turbo run typecheck` ŌĆö zero errors
- [ ] `pnpm turbo run lint` ŌĆö zero warnings or errors
- [ ] `pnpm turbo run test` ŌĆö all tests pass, coverage Ōēź90%
- [ ] `pnpm changeset` ŌĆö changeset created for any user-facing change
- [ ] All new exports have JSDoc (`/** ... */`)
- [ ] No `any` ŌĆö not explicit, not implicit
- [ ] No hardcoded visual values (colors, spacing) ŌĆö only `token:*` references
- [ ] No engine package imports a framework dep (`react`, `react-native`, etc.)
- [ ] All new `ComponentContract`s have all 4 lifecycle states defined
- [ ] `README.md` updated if the public API changed
